== AI integration (KI)

This section describes what the *Cucumber NewsScraper* AI layer can do and how it interacts with your environment.

image::docs/images/diagram-ki-integration.png[AI integration data flow,width=800,align=center]

=== Modes

* *LM Studio REST v1* (recommended with `dev_server`) — `POST /api/v1/chat` with `model`, `input`, and `system_prompt`. Responses may include `output[]` entries of type `message` and optionally `reasoning`. The UI shows only final `message` text, not internal reasoning.
* *OpenAI-compatible* — `POST /v1/chat/completions` with a chat-style payload. Base URL must end with `/v1` (the client appends `/chat/completions`).

Enable *REST via same page origin* in settings when using `scripts/dev_server.py` so the browser does not call LM Studio cross-origin (avoids `OPTIONS` preflight failures on many LM Studio builds).
The *Reasoning* controls are only shown in *LM Studio REST v1* mode. In *OpenAI-compatible* mode that settings block is hidden because this app does not send the LM Studio-specific `reasoning` field there.

=== Model selection (settings)

When a valid server URL is configured, the app loads the model list via native `GET /api/v1/models` first (same-origin proxy with `dev_server`, or direct to LM Studio). Only if that returns `404` does the client retry the older OpenAI-style `GET /v1/models`. You can pick a specific model or choose *No specific choice — server decides* (empty `model` setting): the client then resolves the first suitable LLM from the list (cached per tab in `sessionStorage`). Hover a list entry to see metadata returned by LM Studio (publisher, architecture, max context, format, loaded instance). Models currently loaded in LM Studio are marked with a bullet prefix in the dropdown label.

=== Reasoning settings (LM Studio)

When the native LM Studio model list includes `capabilities.reasoning.allowed_options`, the app filters the *Reasoning level* dropdown to exactly those supported enum values for the currently selected model.

Examples:

* a model that reports `["off","on"]` only shows `Off` and `On`
* a model without `capabilities.reasoning` support keeps only `Off` visible and disables the reasoning toggle

The selected reasoning level and the enabled/disabled state are persisted and restored when the settings dialog is reopened.

If the app has to fall back to OpenAI-style `GET /v1/models`, capability metadata is usually not available. In that case the UI falls back conservatively to `Off` only instead of guessing unsupported reasoning levels.

=== Summary language

* *Site* (`summaryLangMode: site`) — output language is inferred from the article URL (e.g. English for `theverge.com`, German for typical `.de` publishers).
* *Browser* — uses `navigator.language`.

The prompt includes a *source label* (e.g. `heise.de`, `The Verge`) so the model can align tone and context.

=== Caching and batch operations

* Summaries are stored in IndexedDB with a configurable TTL (*Summary cache (days)*; `0` = no expiry).
* *Show summary* uses the cache when fresh; *Regenerate* / *Neu erstellen* sets `forceRefresh` and deletes the cached row first.
* *Summarize all* (header) walks all articles matching the *current category filter*, respects the same cache rules, and uses a configurable concurrency limit and per-request timeout.
* For *new* articles that match the active filter, the app can automatically request summaries when the feed updates (if the model endpoint is reachable).

=== Alternative article links (not LLM-hallucinated URLs)

The model is instructed to produce a summary; *concrete alternative URLs* are fetched separately:

. The client calls `GET /api/search-news` on the dev server with the article headline (Bing News RSS).
. Results are filtered through `AISummarizer.shouldExcludeAlternativeLinkUrl` (drops same-host links and configured news-source domains, plus an optional user domain blocklist).
. Optional: `GET /api/check-url` drops obvious `404`/`410` responses before showing a link.

This keeps “more sources” grounded in real publisher URLs when the dev server is available.

=== Reddit search (AI-enhanced)

The dashboard queries `GET /api/reddit-search?q=…&ai=1` to find related Reddit threads.
When `ai=1` is set and LM Studio is running, the server asks the loaded model to generate 2–3 short English search queries optimized for Reddit (translating German headlines, extracting key terms, using Reddit-typical terminology).
Each AI query is sent to Reddit's search API; results are merged and deduplicated.

If the AI is unavailable (LM Studio not running, model timeout, etc.) the endpoint falls back to the deterministic keyword approach (stopword removal + first-N-token query).

The response includes `ai_enhanced: true` and `ai_queries: [...]` so the client can display which search terms were used.

=== YouTube suggestions

Per-article *YouTube* uses the AI to return structured JSON (search directions and `searchQuery` strings). The UI opens YouTube search result pages — *no YouTube Data API key is required* for that path.

If `YOUTUBE_API_KEY` is set on the server, `GET /api/youtube-search` remains available as an optional legacy integration (`js/youtube-related.js`).

=== Failure modes (short)

* *CORS / Failed to fetch* — use `dev_server` + same-origin REST, or enable CORS in LM Studio, or use the OpenAI-compatible path; see link:troubleshooting.adoc[Troubleshooting].
* *HTTP 400* — wrong model id, or unsupported `reasoning` parameter (the client retries without it when possible).
* *Only reasoning, no message* — change model/settings or use *Regenerate*; the UI surfaces a clear error.
